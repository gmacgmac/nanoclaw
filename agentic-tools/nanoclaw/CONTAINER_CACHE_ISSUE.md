# Docker BuildKit Cache Issue: Container Code Changes Not Appearing

## Problem

You've modified container code (MCP tools, agent-runner, skills) and rebuilt the container, but the changes don't appear when the agent runs inside containers. Even `--no-cache` doesn't help.

## Root Cause

NanoClaw's container architecture has **multiple caching layers** that all need to be cleared:

### 1. Cached agent-runner-src in Session Directories

**This is the most common cause of changes not appearing.**

Each group session has a cached copy of the agent-runner source at `data/sessions/<group>/agent-runner-src/`. The container mounts this cached copy instead of the image's source files. After modifying `container/agent-runner/src/`, you MUST delete these caches:

```bash
rm -rf data/sessions/*/agent-runner-src
```

Without this step, containers continue using the old cached source regardless of image rebuilds.

### 2. Local `dist/` Folder

The `container/agent-runner/dist/` folder contains pre-compiled JavaScript. Docker's `COPY agent-runner/ ./` includes this folder. BuildKit hashes directory contents, so stale `dist/` can be cached even when source files change.

### 3. BuildKit Content-Addressed Cache

Docker BuildKit uses a content-addressed cache where layers are keyed by file content hashes. The cache metadata persists in the builder's volume even after `docker builder prune`. This can cause:

- `COPY agent-runner/` steps to use cached layers from previous builds
- `--no-cache` flag only skips the cache lookup, but doesn't invalidate cached file copies
- The builder volume retains stale file references

### 4. Image Layer Caching

The Dockerfile copies source files early and caches that layer:

```dockerfile
# Step 7 - copies source files (including dist/)
COPY agent-runner/ ./

# Step 8 - builds TypeScript
RUN npm run build
```

If the COPY layer is cached, subsequent builds skip copying new source files entirely.

### 5. Runtime Compilation

The container's entrypoint recompiles TypeScript at runtime:

```bash
#!/bin/bash
set -e
cd /app && npx tsc --outDir /tmp/dist 2>&1 >&2
```

This means the container uses whatever source files exist in `/app/src/` from the image. If the image has stale source files, the compiled output is stale.

### 6. Session Transcript Cache

The SDK stores conversation history in `.jsonl` transcript files. If the agent previously listed tools or described capabilities, that cached text persists in the session and the agent may reference old information instead of re-querying the MCP server.

### 7. SKILL.md Documentation

Files in `container/skills/*/SKILL.md` are synced to the group's session directory at container start. These documents describe available tools and are loaded as context. If they list outdated tools, the agent will report those as available.

## Symptoms

- New MCP tools don't appear
- Modified tool parameters don't work
- Removed tools still appear
- Image ID doesn't change after rebuild
- `docker images nanoclaw-agent:latest` shows same ID before and after build
- Agent reports old tools even after confirming image is correct

## Complete Fix

### Step 1: Delete Cached agent-runner-src (CRITICAL)

```bash
rm -rf data/sessions/*/agent-runner-src
```

**This is the most important step.** Without this, containers continue using old source files regardless of image rebuilds.

### Step 2: Delete Local dist/ Folder

```bash
rm -rf container/agent-runner/dist
```

This folder is generated by `npm run build` and shouldn't be committed. BuildKit may cache its contents.

### Step 3: Prune BuildKit Cache

```bash
docker builder prune -f
```

This clears the content-addressed cache that stores file blobs.

### Step 4: Rebuild from Correct Directory

```bash
./container/build.sh
```

**Important:** This script must run from the repository root, and it internally `cd`s to `container/` before building. Running `docker build .` from the wrong directory will use the wrong build context.

If rebuilding manually:
```bash
cd container && docker build --no-cache -t nanoclaw-agent:latest .
```

### Step 5: Kill Running Containers

```bash
docker ps --filter ancestor=nanoclaw-agent:latest --format "{{.ID}}" | xargs -r docker kill
```

Running containers continue using the old image even after rebuild.

### Step 6: Update SKILL.md Files

Documentation in `container/skills/*/SKILL.md` must match actual MCP tools. These are synced at container start.

```bash
# Check tool definitions in MCP source
grep "server.tool" container/agent-runner/src/ipc-mcp-stdio.ts

# Update SKILL.md to match
```

### Step 7: Clear Session Transcript (Optional)

If the agent's cached tool knowledge is stale, clear the session:

```bash
# Clear session transcript
rm data/sessions/<group>/.claude/projects/-workspace-group/*.jsonl

# Clear session from database so it starts fresh
sqlite3 store/messages.db "UPDATE sessions SET session_id=NULL WHERE group_folder='<group>'"
```

Replace `<group>` with the group folder name (e.g., `dashboard`, `telegram_main`).

```bash
# Full clean rebuild (ALL steps required)

./container/build.sh                        # rebuild image
docker ps --filter ancestor=nanoclaw-agent:latest -q | xargs -r docker kill  # stop old containers
```


## Why `--no-cache` Alone Fails

The `--no-cache` flag tells Docker to not use cached layers during the build. However:

1. **BuildKit's cache is separate** - it stores file content in a content-addressable store
2. **COPY layer hashing** - BuildKit may hash the directory tree differently than file contents
3. **Builder volume persistence** - `docker builder prune` without `-a` only removes dangling cache
4. **Local dist/ folder** - Not affected by `--no-cache`, must be deleted manually

The cache key for COPY steps includes the directory's file listing, which can match even when file contents differ due to how BuildKit hashes directories.

## Prevention

### Before Building Container Code

After modifying `container/agent-runner/src/` or `container/skills/`:

```bash
rm -rf container/agent-runner/dist
docker builder prune -f
./container/build.sh
```

### Check for Uncommitted Changes

The build uses files from the working directory, including uncommitted changes:

```bash
git status container/
git diff container/agent-runner/src/ipc-mcp-stdio.ts
```

### Verify Build Context Size

If the build context is unusually large or small, something may be wrong:

```bash
# Build output shows context size
# "transferring context: 162.91MB" - normal
# "transferring context: 772.86kB" - may indicate cache issue
docker build -t nanoclaw-agent:latest ./container/
```

### Keep SKILL.md Files Synchronized

After adding/removing MCP tools, update the skill documentation:

```bash
# Update container/skills/capabilities/SKILL.md
# Update container/skills/status/SKILL.md
```

## Technical Details

### BuildKit Cache Architecture

```
┌─────────────────────────────────────────────┐
│ Docker BuildKit Builder Volume              │
│                                             │
│  ┌─────────────┐  ┌─────────────────────┐   │
│  │ Content     │  │ Layer Metadata      │   │
│  │ Store       │  │ (cache keys,        │   │
│  │ (blobs)     │  │  timestamps)        │   │
│  └─────────────┘  └─────────────────────┘   │
│                                             │
│  docker builder prune → removes dangling    │
│  docker builder prune -a → removes all     │
└─────────────────────────────────────────────┘
```

### Container Runtime Flow

```
docker run nanoclaw-agent:latest
        │
        ▼
┌─────────────────────┐
│ entrypoint.sh       │
│                     │
│ 1. npx tsc          │ ← Compiles /app/src/*.ts
│    --outDir /tmp/dist │
│                     │
│ 2. node index.js    │ ← Runs compiled code
│    < input.json     │
└─────────────────────┘
        │
        ▼
   /tmp/dist/
   ├── index.js
   └── ipc-mcp-stdio.js  ← MCP tools loaded here
```

The key insight: **the image's `/app/src/` directory must have current source files**. The runtime compilation doesn't bypass the image's file system.

### Skill Synchronization Flow

```
Container Start
        │
        ▼
┌─────────────────────────────────────────────┐
│ src/container-runner.ts                     │
│                                             │
│ Sync container/skills/* →                   │
│   data/sessions/<group>/.claude/skills/    │
│                                             │
│ (Per-group filtering via containerConfig)  │
└─────────────────────────────────────────────┘
        │
        ▼
   Agent loads SKILL.md files as context
   (Tools described in docs must match actual MCP)
```

## When to Rebuild What

| What Changed | Action |
|--------------|--------|
| `src/` (host code) | `npm run build` + restart service |
| `container/agent-runner/src/` | `rm -rf dist && docker builder prune -f && ./container/build.sh` |
| `container/skills/` | Update SKILL.md, rebuild (synced at runtime, but rebuild ensures consistency) |
| `container/Dockerfile` | `docker builder prune -f && ./container/build.sh` |
| Both host and container | Both actions, plus restart service |
